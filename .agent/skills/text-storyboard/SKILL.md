# Text Storyboard

Use this skill to create or normalize a text storyboard Markdown file from scripts, prose,
legacy cut lists, or raw scene notes.

This is a ScarletEchoes project-local skill. Resolve project files relative to `{PROJECT_ROOT}`, the repository root that contains `README.md`, `common/`, `stories/`, and `.agent/`.

This skill owns the text-conte stage. `gpt-image2-storyboard` must receive an already prepared
text storyboard and should not create or rewrite the text storyboard except for reading it.

## Core Rule

Every text-storyboard scene must be 15 seconds or shorter because one scene maps to one video-generation request. Split any beat longer than 15 seconds at a clear action, dialogue, camera, or emotional turn. Never output `- 時間:` with a duration greater than `15秒`.

All file paths written into the text storyboard must be relative to the target story's anime production folder, `{ANIME_ROOT}` (`{STORY_ROOT}/02_Anime`), not relative to the Markdown file location. For example, write `references/alvina_sheet.png`, `background/講堂_1.png`, or `storyboards/01/scene_034_stage_sketch.png`.

When reading existing project files, normalize paths by context before checking files or passing references:

- Paths beginning with `{PROJECT_ROOT}`, `{STORY_ROOT}`, or `{ANIME_ROOT}`: expand the variable.
- Absolute paths: use as-is.
- `stories/...` or `common/...`: resolve from `{PROJECT_ROOT}`.
- `02_Anime/...`: resolve from `{STORY_ROOT}`.
- `references/...`, `background/...`, `storyboards/...`, or `designs/...`: resolve from `{ANIME_ROOT}`.
- Anything else is ambiguous; do not resolve it relative to the Markdown file by guesswork.

This means `REFERENCE_MAP.md` may contain legacy project-root-relative paths such as `stories/01_緋色の魔法遣い/02_Anime/references/alvina_sheet.png`, while the text storyboard must still write the same asset as `references/alvina_sheet.png`.

Use project-root-relative paths for repository rule files and story roots:

- `common/絵コンテルール.md`
- `common/動画生成ルール.md`
- `stories/[Story_Title]/02_Anime`

When this skill needs to generate a stage sketch or scene visual image, use the current runtime's native image-generation tool directly. In Codex, this means the native GPT Image2-capable image generation route; in Gemini-based runtimes, this means the native Nano Banana Pro-capable route or equivalent. Stage sketch creation is a two-pass workflow:

1. Generate a prompt-only rough `LOCATION / BLOCKING OVERVIEW` sketch with the native image-generation tool, optimized for overhead or high-angle spatial continuity.
2. Save that draft sketch, load it with `view_image` or the runtime's equivalent visible-image mechanism together with the resolved character and background reference images, then run a second native image-generation edit pass that preserves the map layout while replacing/aligning named characters and the stage with the references.

This two-pass native-generation workflow is the project-local route for stage sketches. It relies on conversation-visible image context: before the second pass, the draft sketch and every required reference image must be visibly loaded in the conversation.

If the runtime's native image-generation route cannot be used for stage-sketch or scene-visual drawing, stop as blocked and report the missing capability. Do not fall back to image API calls, CLI wrappers, PowerShell/Python scripts, or custom drawing/compositing code for these drawing tasks.

Do not treat the first prompt-only draft as the completed stage sketch when named character or background continuity is required. The completed stage sketch is the second-pass consistency image.

Stage sketches and scene visual images must be generated by the runtime's native image-generation tool, not by code compositing.

- Before generating a sketch, read `{ANIME_ROOT}/references/REFERENCE_MAP.md` and `{ANIME_ROOT}/background/REFERENCE_MAP.md`.
- Resolve all named characters, monsters, props, and locations in the target scene through those maps.
- For the second pass, load the prompt-only draft sketch, resolved character reference images, relevant face/reference variants, background reference images, and previous stage sketch when continuing a stage as visible image context before calling the native image-generation tool.
- If a character or background is not defined in the maps, do not guess from filenames. Mark it as unresolved and either add a reference-map TODO in the text storyboard or stop if the missing reference is required for visual continuity.
- The second-pass sketch prompt must explicitly state which visible image is used for each role: `draft_stage_sketch`, `character_reference`, `background_reference`, `scene_visual`, `prior_stage_sketch`, or `prop_reference`.
- Main characters in the stage sketch must reflect their reference images enough to keep identity and placement consistent. Generic mobs must stay generic and must not inherit protagonist traits.
- Do not create stage sketches by combining, pasting, tracing, cropping, or compositing reference images with custom code or image-processing scripts. Non-creative file operations such as copying, renaming, and verification are allowed only after the native image-generation tool has generated the image.

Always preserve dialogue metadata in the text storyboard, but write it inside the corresponding `CUT PLAN` item rather than as a scene-level dialogue section.

- Keep `セリフ:` inside every CUT PLAN item, even when it is `なし` or `ナレーションなし`.
- Preserve speaker names and quoted dialogue exactly unless the user asks for adaptation.
- Preserve monologue markers such as `M` and narration labels.
- Dialogue is metadata for downstream video prompts and lip-sync.
- Dialogue must not be drawn into storyboard worksheet image cells unless the user explicitly asks and the project rules allow it.

## Workflow

1. Read the source script, prose, legacy text conte, or user-provided scene notes.
2. Read project rules when available:
   - `common/絵コンテルール.md`
   - `common/動画生成ルール.md`
   - `{ANIME_ROOT}/references/REFERENCE_MAP.md`
   - `{ANIME_ROOT}/background/REFERENCE_MAP.md`
3. Identify the target unit:
   - episode, scene range, or cut range
   - output path and filename requested by the user
   - storyboard output folder where both text storyboard and stage sketches will be saved
4. Extract or infer for each scene:
   - scene number
   - time / duration, capped at 15 seconds per scene
   - content summary
   - dialogue as structured metadata inside the matching `CUT PLAN` item
   - location / environment state
   - characters
   - camera
   - screen description
   - direction
   - SE / sound notes
   - continuity notes for fixed characters, semi-fixed groups, and background mobs
5. Track stage continuity while building the text storyboard.
   - When a new stage or materially changed stage state first appears, create a stage sketch before finalizing the text storyboard.
   - Use the two-pass stage-sketch flow:
     1. Prompt-only layout pass: generate a rough overhead or high-angle `LOCATION / BLOCKING OVERVIEW` / `BLOCKING MAP` with the native image-generation tool. Prioritize stage geography, desks, doors, windows, fixed landmarks, character labels/icons, mob density, and movement arrows over character likeness.
     2. Reference consistency pass: save the draft, load it with `view_image` or the runtime's equivalent visible-image mechanism together with every resolved character/background/prop reference, then call the native image-generation tool again to preserve the layout while aligning named character identity and stage details to the references.
   - Before the second pass, resolve all applicable character and background references from `REFERENCE_MAP.md` and load the actual image files into the conversation context. Do not stop at the first prompt-only draft when references are required.
   - If the scene continues an existing stage, attach the previous stage sketch as `prior_stage_sketch` so seats, standing positions, facing directions, crowd density, and movement paths remain stable.
   - Save the sketch PNG in the storyboard output folder, using a predictable name such as `scene_034_stage_sketch.png` or `stage_sketch_講堂_01.png`.
   - When recording the sketch in the text storyboard, convert the saved file path to an `{ANIME_ROOT}`-relative path such as `storyboards/01/scene_034_stage_sketch.png`.
   - The prompt-only draft sketch must include the fixed stage layout, main character positions, facing directions, seats/standing marks, supporting figures, mob density, entrances/exits/windows/furniture/key props, movement paths, and simple labels/legend when useful.
   - The final second-pass sketch must preserve the draft's map format, camera angle, labels/legend, arrows, entrances/exits, seating banks, and blocking logic unless the references reveal a necessary correction.
   - Main character silhouettes and visible traits must match attached references closely enough to distinguish Alvina, Justina, Patrona, Letizia, and other named characters in downstream storyboard generation.
   - Make nearby mobs visually generic. Do not let protagonist traits propagate to background students or crowds.
   - If the same stage continues, reuse the same sketch filename in every later scene until the stage state changes.
6. Convert each scene into `CUT PLAN` visual beats.
   - Keep each scene duration at or below 15 seconds.
   - If the source beat is longer than 15 seconds, split it into multiple numbered scenes before writing `CUT PLAN`.
   - Keep cut times continuous and equal to scene duration.
   - Do not divide time evenly by default; assign each cut duration based on dialogue length, acting beats, reaction time, held tension, quick insert shots, and the narrative purpose of the cut.
   - Keep visual actions concise.
   - Write dialogue inside the corresponding cut only; use it to infer acting and mouth movement without creating a separate dialogue section.
7. Add `舞台スケッチ:` to every scene that uses a fixed or recurring stage.
   - Use a path relative to `{ANIME_ROOT}`, even when the Markdown file is saved inside `storyboards/<episode>/`.
   - If no sketch is needed because the scene is abstract or has no stable stage, write `舞台スケッチ: なし`.
8. Save the normalized text storyboard Markdown.
9. Then hand off to `gpt-image2-storyboard` to generate storyboard worksheet images from the saved Markdown.

## Required Output Format

Use this structure for each scene:

```markdown
## シーン 37

- 時間: 08:21-08:36 / 15秒
- 内容: アルヴィナがドラゴンについて質問し、教室が凍る。
- 場所状態: 講義室継続。春光が冷たく感じられる。
- 舞台スケッチ: storyboards/01/scene_034_stage_sketch.png
- 登場人物: アルヴィナ、パトロナ、ユスティナ、生徒群。
- 参照ロール:
  - character_reference:
    - アルヴィナ: references/alvina_sheet.png
    - パトロナ: references/patrona_sheet.png
    - ユスティナ: references/justina_sheet.png
  - background_reference:
    - 講堂: background/講堂_2.png
  - stage_sketch: storyboards/01/scene_034_stage_sketch.png
- カメラ: ALVINA STAND / ROOM FREEZE
- 画面: 挙手するアルヴィナ。周囲の視線が一斉に集まる。
- 演出: 音をすっと引き、囁きだけを残す。
- SE: 椅子の軋み、囁き。
- CUT PLAN:
  - CUT1 [00:00-00:02] MEDIUM / STAND:
    - 目的: アルヴィナが立つ瞬間を短く提示する。
    - 画面: アルヴィナが挙手して立ち上がる。
    - セリフ: なし
  - CUT2 [00:02-00:08] MEDIUM / SPEAK:
    - 目的: 主発話を収め、周囲が固まるきっかけにする。
    - 画面: アルヴィナがまっすぐ質問する。
    - セリフ: アルヴィナ「質問！ 《ドラゴン》は魔物に分類されますか？」
  - CUT3 [00:08-00:11] WIDE / FREEZE:
    - 目的: 発話後の沈黙をタメる。
    - 画面: 教室全体の視線が集まる。
    - セリフ: なし
  - CUT4 [00:11-00:15] REACTION / WHISPER:
    - 目的: 生徒たちのざわめきへ広げる。
    - 画面: 生徒たちが小声でざわめく。
    - セリフ: 生徒たち「ドラゴン……？」
```

## Handoff Contract

When the next step is storyboard image generation:

- Pass the saved Markdown path to `gpt-image2-storyboard`.
- Ensure every recurring concrete stage has a `舞台スケッチ:` field that points to a saved sketch PNG using an `{ANIME_ROOT}`-relative path.
- Tell `gpt-image2-storyboard` to use `舞台スケッチ:` as an actual image reference for all scenes sharing that stage.
- Tell `gpt-image2-storyboard` to read `セリフ` from each `CUT PLAN` item as staging context and not to write dialogue into worksheet cells.
- Do not ask `gpt-image2-storyboard` to create or normalize the text storyboard.

When the next step is video prompt generation:

- Pass the saved Markdown path and storyboard worksheet images to `video-prompt-builder`.
- Ensure dialogue from each `CUT PLAN` item's `セリフ:` is copied into the corresponding video-prompt cut instructions, not into a separate dialogue section.
